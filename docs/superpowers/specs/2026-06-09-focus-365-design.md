# Focus 365 — Diseño del sistema

**Fecha:** 2026-06-09
**Estado:** Aprobado (diseño) · pendiente plan de implementación
**Autor:** Gustavo (con Claude)

## 1. Visión

Focus 365 es un sistema personal de mejora 360° que evoluciona el vault de Obsidian "Capitanes de Dios" hacia una **web app real, escalable y automatizada**. Gestiona cinco frentes de la vida en un solo lugar y usa un **asistente de IA con contexto completo** para mantener al usuario enganchado y avanzando.

**Dimensiones / módulos:**
1. **Check-in diario** — ánimo, energía, disciplina del día.
2. **Finanzas** — captura nativa de gastos/ingresos + superávit por ciclo de pago.
3. **Entrenamiento** — rutinas y registro físico.
4. **Mente / Disciplina** — hábitos, rachas, retos (p. ej. "Reto 21 días").
5. **Metas** — objetivos por dimensión con seguimiento.
6. **Asistente IA** — análisis proactivo y asistencia sobre todos los datos.

**Principios:**
- **Personal primero**, pero arquitectura lista para escalar a producto multi-usuario.
- **Que trabaje sola**: automatización, insights proactivos, gamificación (rachas) para enganchar.
- **Build-with-me**: el usuario hace commits, aprende y modifica el código.

## 2. Stack y arquitectura

**Monorepo, todo dockerizado** (portabilidad). Despliegue en VPS KVM 4 vía **Coolify** (3 servicios desde un mismo repo).

| Capa | Tecnología |
|------|-----------|
| Frontend | React 18 + Vite + **TanStack Router** + **TanStack Query** + Tailwind + shadcn/ui — **SPA** servida por nginx |
| Backend | **Go** — chi (router), pgx (driver), sqlc (queries tipadas), goose (migraciones), JWT + bcrypt (auth), validator |
| Base de datos | **PostgreSQL 16** |
| IA | **Groq** (llama-3.3-70b-versatile), clave server-side únicamente |
| Infra | Docker Compose local · Coolify en VPS · imágenes distroless (api) y nginx (web) |

**Decisión clave — SPA, no TanStack Start:** Start incluye capa de servidor que se solaparía con el backend Go. Mantenemos React como SPA pura (Router + Query) y todo el servidor en Go: una sola fuente de verdad para API y auth.

**Estructura del monorepo:**
```
focus-365/
├── api/                 # Go
│   ├── cmd/server/      # main.go
│   ├── internal/server/ # router, middleware, handlers
│   ├── db/migrations/   # goose
│   └── db/queries/      # sqlc
├── web/                 # React/Vite SPA
│   └── src/
├── docker-compose.yml   # db + api + web
└── docs/
```

**Flujo de datos:** SPA → `/api/v1/*` (proxy a Go en dev; nginx + Coolify en prod) → handlers Go → sqlc/pgx → Postgres. Toda query está **scoped por `user_id`** desde el día uno.

## 3. Modelo de datos (núcleo)

Postgres, UUIDs (`gen_random_uuid()`), timestamps en UTC. Todas las tablas de dominio llevan `user_id` (FK → `users`).

- **users** — id, email (unique), password_hash, name, created_at.
- **check_ins** — id, user_id, date, mood (1-10), energy (1-10), discipline (1-10), note, created_at. (único por user_id+date)
- **transactions** — id, user_id, type (income|expense|transfer), amount, occurred_on, category, remark, source (manual|import), external_id (nullable), created_at.
- **financial_months** (derivado/cache) — user_id, cycle_label, income, expense, net, status (pendiente|verde|rojo). Agrupa por **ciclo de pago**, no calendario.
- **workouts** — id, user_id, date, type, detail/exercises (jsonb), created_at.
- **habits** + **habit_logs** — definición de hábito/reto y registro diario para calcular rachas.
- **goals** — id, user_id, dimension, title, status (active|done|paused), progress, deadline, created_at.
- **ai_insights** — id, user_id, generated_at, kind (proactive|on_demand), content, context_snapshot (jsonb).

### Ciclo de pago (finanzas)
La nómina cae el **último día hábil del mes** (si cae sábado → viernes; domingo → viernes) y financia el mes siguiente. Un movimiento pertenece al mes financiero que sigue al payday más reciente ≤ su fecha. Ej.: nómina del 29 may → "Junio" (29 may–28 jun). **Meta:** ningún mes (jun–nov 2026) cierra en rojo.

### Importación desde la app de gastos (one-time / asistida)
La API externa (`money.quhou123.com`, `getTransactions`) usa token JWT efímero (1 h) enviado como campo de formulario. Se usa para **importar histórico** a Postgres; el flujo diario es **captura nativa**. No se guarda ningún secreto en el repo.

## 4. Auth y API

- Registro/login con **bcrypt**.
- **JWT access** (~15 min, header `Authorization: Bearer`) + **refresh token** en cookie `httpOnly`.
- Middleware chi inyecta `user_id` en el contexto; todos los handlers filtran por él.
- Versionado: `/api/v1/...`. Health: `GET /api/v1/health`.
- Multi-usuario listo desde el inicio aunque el uso sea personal.

## 5. Asistente IA (Groq)

- **Contexto completo:** el backend arma un snapshot (finanzas del ciclo, racha actual, check-in del día, metas) y lo envía a Groq.
- **Dos modos:**
  1. **On-demand** — el usuario pregunta ("¿cómo voy en junio?") y la IA responde con sus datos.
  2. **Proactivo (preferido)** — la app genera **insights del día** sin que el usuario pregunte: detecta patrones (gasto hormiga subiendo, racha en riesgo, ánimo alto = aprovecha) y los muestra arriba del dashboard. *(Opción 2 elegida como prioridad.)*
- Clave Groq **solo en el servidor**; el frontend nunca la ve.

## 6. UI / Navegación

- **Shell:** Top bar + **dashboard de tarjetas** ("centro de mando"). (Opción B elegida.)
- **Dashboard (centro de mando):**
  - **Insight de IA full-width arriba y grande** (banda destacada con CTAs).
  - Saludo + fecha + nº de dimensiones en marcha.
  - Rejilla de tarjetas: **Racha**, **Superávit del ciclo**, **Ánimo/Energía** (barras), **Check-in de hoy**, **Entreno de hoy**, **Metas activas**.
- **Estilo visual: B · Warm Discipline** — oscuro cálido (carbón + ámbar/oro), íntimo y motivacional (sensación de diario premium). Tipografía Inter. Acentos: ámbar IA, verde dinero, naranja-cálido racha. Posible modo claro más adelante.
